## One-Command Capture

Use the capture tool:

```powershell
.\tools\collect_thor_debug.ps1 -Label odin-black-screen
```

It writes an ignored local folder under `debug-captures/` with:

- full recent logcat
- filtered crash/emulator/GPU logcat
- persistent `RPCSX.log`, `RPCSX.old.log`, `RPCSX.log.gz`, and `TTY.log`
- `config.yml`, `games.yml`, `patch_config.yml`, `rpcn.yml`
- activity/window/meminfo/gfxinfo/thermal snapshots
- cache and installed-game summaries

For a clean repro:

```powershell
.\tools\collect_thor_debug.ps1 -Prepare -Launch -Label folklore-black-screen
# reproduce the issue on the Thor
.\tools\collect_thor_debug.ps1 -Label folklore-black-screen
```

`-Prepare` clears logcat so the next capture starts near the repro. Do not use it after the crash, because it deletes the evidence.

## Live Stream Workflow

For the deeper loop where the user plays on Thor and Codex watches from the workstation, use a streaming session:

```powershell
.\tools\start_thor_debug_stream.ps1 -ClearLogcat -Launch -Label odin-black-screen
.\tools\summarize_thor_debug_stream.ps1 -Latest
.\tools\stop_thor_debug_stream.ps1 -Latest
```

The stream creates an ignored `debug-captures/*-stream/` folder with:

- `logcat-live.txt`: continuous Android logcat
- `rpcsx-live-tail.txt`: repeated tail of the persistent emulator log
- `live-summary/now-*.txt`: refreshed PID, focus, top threads, memory, thermal, cache, and game-data snapshots
- `summary-latest.md`: quick diagnosis generated by the summary script
- `final/`: pulled logs and final state after stopping the stream

This is the practical agent loop:

1. Start the stream and launch the app.
2. User reproduces the issue on Thor.
3. Codex runs the summary script repeatedly while the stream is active.
4. If a clear code/config issue appears, Codex patches and rebuilds.
5. Stop the stream after the repro so the final emulator log and config snapshot are preserved.

## Debug Buckets

Use three buckets. They need different fixes.

### 1. Native/app crash

Signs:

- app returns to launcher
- `pidof net.rpcsx.easy` goes empty
- logcat has `Fatal signal`, `SIGSEGV`, `SIGILL`, `SIGABRT`, `FATAL EXCEPTION`, or tombstone output

First questions:

- Did it die in `librpcsx-android.so`, Vulkan/Turnip, LLVM JIT code, or Android UI/JNI?
- Was the custom GPU driver active?
- Was the title using old `cortex-x3` cache, new `cortex-a78` cache, or compiling live?

### 2. Black screen but app alive

Signs:

- `RPCSXActivity` is still focused
- process is alive
- logs continue with PPU/SPU/RSX/syscall activity
- audio may continue

First questions:

- Is it actually installing game data or loading a massive asset?
- Did recommended settings create a bad per-game config?
- Did the game need a PS3 system dialog that our Android overlay path is not displaying?
- Is RSX presenting frames, or are PPU/SPU threads waiting forever?

### 3. Frozen after settings/core change

Signs:

- same game used to boot
- failure follows a fork change, driver change, cache move, cheat toggle, or fast-forward toggle

First questions:

- Try default GPU driver.
- Disable cheats for that title.
- Disable recommended per-game config.
- Clear only that title's PPU/SPU cache after LLVM CPU target changes.
- Confirm `Use LLVM CPU: cortex-a78` in `config.yml`.

## Next App Improvements

Best low-friction additions:

1. Add a `Debug Bundle` button in Settings and Game Detail that runs the same capture intent from the app side or at least opens instructions with the exact title ID.
2. Add a per-game `Boot Problem Mode` toggle that temporarily disables cheats, recommended config, fast-forward, custom GPU driver, and risky overlays for one launch.
3. Add visible boot-state breadcrumbs: title ID, boot path, active GPU driver, LLVM CPU target, cache entry count, whether the game is reading `/dev_bdvd` or writing `/dev_hdd0/game`.
4. Add a "black screen detector" overlay after 60 seconds with no presented frame but continuing PPU activity: show `Still booting/installing`, `Compiling`, `Waiting on dialog`, or `No RSX frame yet`.
5. Keep a rolling mini event log in app storage: launch, title ID, driver, config source, first RSX frame, first audio stream, first native Home Menu open, crash/stop.

## BLUS31386 PPU Compile Crash

The first live stream test on Thor caught a real native crash during PPU LLVM compilation:

- Title/cache path: `BLUS31386`
- Crash time: about 4 minutes into PPU precompilation
- Thread: `PPUW.1.1`
- Native signal: `SIGABRT`
- Core failure: `llvm::report_bad_alloc_error` while `llvm::RuntimeDyld` / `MCJIT` was loading generated code
- RPCSX fatal log: `memory_commit(... errno=12=Out of memory)` at `rpcs3/util/vm_native.cpp:332`
- Device thermal was high during compile, with fan logs around 84-85 C before the crash

Thor config at the time had:

- `LLVM Precompilation: true`
- `Max LLVM Compile Threads: 4`
- visible PPU workers around `PPUW.1.1` through `PPUW.1.3`

Working fix: change Thor startup defaults to avoid full first-boot PPU precompilation. For this fork, prefer `LLVM Precompilation: false` and `Max LLVM Compile Threads: 2` by default, then let games compile on demand or use a future guarded cache builder that can adapt thread count after measuring memory pressure. This is slower/stutterier than full precompile, but it should avoid first-boot native aborts on memory-heavy titles.
